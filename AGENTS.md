# nebius-serverless-tuner — guide for the coding agent

The always-loaded contract for how to work in this repo. The per-task prompt and the implementation
plan always take precedence where they conflict.

## What this is

An agentic AI-performance-engineering tool: it deploys an open model as a Nebius Serverless
**Endpoint** (vLLM, OpenAI-compatible), benchmarks it via a Serverless **Job**, then an **agent**
tunes the serving config to minimise cost-per-token and emits a report. Entry for the Nebius
Serverless AI Builders Challenge.

The whole loop runs behind injected seams (`CliRunner`, `Client`, `Generate`) so every unit is
unit-tested fully offline — **no network, no GPU, no billing in tests.**

## How you work here (build discipline)

Built **one task at a time** from `docs/superpowers/plans/2026-06-16-serverless-tuner.md`. For each task:

1. **One task per run.** Do exactly the named task — nothing adjacent.
2. **Only the named files.** Each task lists the only files it may touch.
3. **TDD, strictly.** Write the failing test first → run it → confirm it fails for the stated
   reason → implement until green. The plan gives the exact test + implementation code — follow it.
4. **Verify.** Run the exact test command in the task and confirm the expected pass count. Never
   leave the suite red.
5. **Do NOT commit.** Leave changes in the working tree; the owner's tooling reviews and commits.
6. **No scope creep.** If a task seems to need an unnamed file, stop and surface it.

## Conventions

- Python 3.11+, `pytest`. Run tests with `.venv/bin/pytest`.
- **No AI attribution** anywhere — not in code, comments, docstrings, or anything you write. No
  "generated by", no agent names. Write as a normal engineer would.
- Type/field names are fixed in `src/tuner/config.py` (Task 1) — never redefine them.
